리팩토링-교보재-1

오늘은 불필요한 지역변수를 사용한 메소드를 다이어트 시키는걸 목표로 하겠습니다(이 코드가 실화냐고 물어보지 말아주세요. 저도 슬픕니다.).
각설하겠습니다.

Given

먼저 타겟 코드입니다(쓸데없는 주석은 모두 제거…T.T). 함수 바디만 기술했습니다.

잘 돌아가는 코드지만,(중요!)

When

일단 코드를 풀어보면.

  • return 할 배열을 param으로 선언.
    • 네이밍 센스
    • 꼭 필요한 변수인가?
  • params 변수에 다시 split값을 할당
    • 아마도 함수는 재사용 불가할 듯(side effect)
  • for문을 사용해야 하기 때문에 size를 선언
    • 가독성(타인을 위한 배려)을… 위해 선언했다면 인정…
    • for문 안에 변수없이 사용할 수 없는가? 이 경우 가독성을 해치는지?(아마도 가독성이 향상될 듯)
  • for 안에서 사용할 변수 선언
    • size 변수와 마찬가지
  • if 문에서 indexOf를 사용
    • 위 문제들과 마찬가지로 가독성문제
    • 물론 해당 코드를 읽는 사람들이 익숙했다면 인정…
  • if 문 안에서 결과를 위한 작업 진행(핵심 코드)
    • 같은 코드가 중복되어 있음 -> 같은 일을 두 번 해야함

Then

근데 사실 코드를 잘 읽어보면 무슨일을 하는지도 알것같고, 의미도 유추 가능할 것 같습니다.
그래서 실무에선 그냥 넘어간 코드입니다. 그럼 어떻게 하면 조금은 더 깔끔한 코드로 만들 수 있을까요? 조금 고민한 끝에 아래 두 코드를 작성했습니다.

만족스러운(지극히 개인적) 결과라고 생각합니다. 1번의 경우 몇몇 문제가 해결 되었지만 여전히 뭔가 냄새가 납니다(물론 #1이든 #2든). 2번에서는 냄새를 조금 더 지우기 위해서 노력했습니다.

  • #1
    • 기존 코드(params) 사용
    • 여전히 리턴할 값을 담을 무언가가 필요함 -> result
    • 지나치게 장황한 변수명은 아닌가 생각함 -> keyValueArray
  • #2
    • 여전히 기존 코드(params) 사용 -> params 변수로 받을것인가?

params에 대해서 논할 필요가 있겠지만 무작정 한 줄에 이어 붙이기 보다는 한 번 끊고 변수로 설명 한다는 의미로 생각했습니다.
사실 제가 코드의 양(줄 수라던가, 함수의 물리적 (혹은 논리적) 크기)을 다룰 레벨은 아니고 그래도 최대한 여러 원칙들(최대한 변수는 불변으로, 불필요한 변수나 중복을 피하자 등)을 지키고자 노력할려고 노력하는 스타일입니다. 아는만큼 보인다고 제가 아는정도에 한해 작성한 코드이고 앞으로 더 많이 공부하고 더 많이 알게되면 쬐끔은 멋진 그리고 간결하면서 할 일 다하는 코드를 작성하고 싶습니다.

다음에는 Spring Framework - 필드 뭣이 중헌디?! 로 떠들어 보겠습니다.

감사합니다!

끗!

도커로-이사가기-2

이번 글에서는 docker-compose를 사용해 사용자에게 보여줄 문구를 서버에서 받아와 표시하도록 하는 작업을 목표로 하겠습니다. 어디까지나 테스트 용도이고 실제 배포에서 사용하려면 더 세부적인 설정이 필요하겠죠?! 현재 목표는 로컬에서 구동이 가능한 앱을 만드는 것입니다. 물론 악마는 언제나 세부사항에 있습니다…ㅠㅠ

일단 작업을 위해 정리를 하면

  • 화면은 Vue.js(Vue cli)를 이용
  • 서버는 Spring-boot
  • 데이터베이스는 MySQL
  • 화면 <-> 서버를 호출 - CORS 해제(세부 사항은 각자ㅠㅠ)
  • 도커 프로세스간 호출 설정

정도면 한 회전 정도는 개발이 가능할 것 같습니다.

추후에 하고싶은 것:

  • CI/CD 도입
  • 무중단 배포 적용

Docker-Compose가 뭔데?

간단하게 docker-compose는 여러 도커 앱의 정의 및 실행을 도와주는 도구입니다. 도커 앱간의 상관 관계(어떤 의미로는 도커 프로세스의 구동 순서 쯤?)라던가 서로 참조하는 등의 작업을 쉽게 해주는 역할도 포함해서요.

Front, Server, Db: Docker-compose 설정

그럼 먼저 docker-compose.yml 파일 내용을 보겠습니다.
docker-compose 파일 버전 확인

docker-compose.yml

먼저 폴더 구조는

  • / (#project root)
  • L–server
  • L–my-project(client)
    와 같고 server에 docker-compose.yml을 위치시켰습니다(각자 환경에 맞게 옮기셔도 무방).
  1. versions: docker-compose 파일 버전을 명시합니다(보통 2 또는 3을 사용하는데 3에서는 2에서 많이 쓰던 links 등의 옵션이 deprecated 됨).
  2. 도커가 구동할 서비스(앱)은 총 3가지로 DB, SERVER, CLIENT로 구성됨
  3. db - 이미지는 mysql:5.7버전을 사용, 그외 환경변수 설정
  4. server - context는 작업 폴더로 생각하면 쉬움
    : spring-boot를 이용한 도커 앱으로 docker-compose.yml 파일과 함께있는 Dockerfile을 이용해 이미지를 생성해 사용함
    : 포트 - 호스트의 8080과 도커 앱의 8080을 연결
    : 그외 환경변수 설정 - DB_HOST의 경우 도커내에서 도커 service 이름으로 참조가 가능하므로 도커 service 이름을 전달해줘 spring-boot가 인식할 수 있도록 함
    : depends_on - 도커 서비스인 db 가 구동이 된 후에 service를 구동하도록 의존을 설정
  5. client - context를 변경해주고 Dockerfile-dev 파일을 이용해 이미지 생성

이렇게 docker-compose 파일을 생성하고, server용 Dockerfile을 생성합니다(db의 경우 compose에서 하는일이 전부이고 client는 이전 글에서 생성함).

Dockerfile - server

간단합니다.

  • openjdk(권장) 이미지로부터
  • 볼륨 생성
  • 빌드된 app.jar파일을
  • 실행

정말 간단합니다.

아주 기본적인 명령이기 때문에 몇 가지 옵션을 소개하면서 글을 마치도록 하겠습니다.

Vue.js + Docker

도커로 무언가를 하는 행위를 도커라이즈라고 한다고도 하지만 잘 모르니 ㅎㅎ 패쓰!
Vue.js 는 Vue cli로 설치하도록 하겠습니다.

이렇게 쓰라고 하는군요!

그럼 아래처럼 생성!

생성시에 뭘 막 물어보는데 저는 예스맨이니까 다 Y로 해줍니다(많이 접해보지 못한 e2e테스트 등도 경험해 볼겸!).

8080포트에 Vue.js로고와 함께 Vue 프로젝트가 생성되었습니다. 이얏호!

그럼 이제 본격적으로 도커 파일을 만들어보겠습니다.

  • 1라인 FROM~: 여기는 기본이되는 이미지를 지정해줍니다.
  • 4라인 ADD~: 여긴 제(Vue cli)가 작성한 package.json파일을 도커에서도 참조하도록 도커내의 /tmp폴더에 추가해줍니다.
  • 5, 6라인 RUN~: 도커 내에서 실행할 명령입니다.

여기까진 아마도 잘 안바뀌겠죠…? 지금 보면 아마도 1라인의 노드 이미지 버전이 바뀐다던가 하는 정도가 아니면요?

  • WORKDIR: 도커안에서 cd /usr/src/app을 한 효과를 줍니다. 작업 디렉토리가 변경됩니다.
  • RUN: 도커에서 실행할 명령들
  • ENV: 환경변수를 만들어질 도커 이미지에 설정
  • EXPOSE: 도커안에서 밖과 소통할 수 있는 창구 포트
  • CMD: 도커안에서 실행할 명령과 변수들

이제 위 Dockerfile을 아까 생성된 Vue 프로젝트의 루트에 넣어줍니다(Dockerfile 안에서는 상대경로 참조가 안되는것 같아요).

도커 만나기

먼저 도커는

  1. 이미지를 생성
  2. 이미지를 구동
  3. 이미지 재 생성
  4. 이미지 재 구동

정도의 순서로 실행된다고 간단하게 알고 넘어가겠습니다.

제일 먼저 도커의 이미지를 생성하고 생성된 도커 이미지에 별칭(Tag)을 달아주는 일을 하겠습니다.

현재 폴더의 도커 파일을 이미지로 생성해주고 태그(이름)는 vue-project로 지정해줍니다.

생성된 이미지는 images 명령을 통해 보실 수 있습니다.

그리고 생성된 이미지를 통해 run 명령으로 구동시켜 줍니다.

  • -d: 백그라운드에서 구동
  • –rm: 구동 중지 시 프로세스 목록에서 삭제
  • -p: 호스트 8080포트와 도커 8080포트를 연결

위 명령을 통해 도커id(d72~)로 실행되고 있고 접속해보니다. http://localhost:8080

잘 실행되고 있습니다.
실행중인 도커를 멈추면 프로세스 목록에서도 사라집니다.

멈출때는 아이디에 앞 몇자리만 입력해도 도커가 인식합니다.
여러 프로세스를 종료할 때 더 간단하게 멈출 수 있고 아예 명령을 연결하면 실행중인 모든 프로세스/종료된 목록을 중지/삭제시킬 수 있습니다.

오늘은 도커이미지 생성 구동 실행 종료 삭제를 알아봤습니다.
곧 docker-compose와 여러 도커 프로세스를 연결하고 한번에 올렸다 내렸다 하는 글을 쓰도록 하겠습니다.

별 내용도 없는데 글이 기네요.ㅠㅠ 곧 또 포스팅 해보도록 하겠습니다. 사정상 급 마무리.

안녕 깃헙 페이지?!

Github.io 를 이용하고 싶어 지킬을 알아보고 설치하고 방치한지 1년이 지났는데…
우연히 Hexo를 발견했고, 지킬이 어려운 저같은 쫄병에게 아주 쉽게 다가와줬습니다.
일단 얼마나 쉽게 했는지를 써 내려 간 후 앞으로

  • 도커를 이용한 Vue.js + Spring boot 기본 프로젝트 생성을 삽질하면서 연재하도록 할 예정입니다.

오늘은 Hexo 설치부터 배포까지…

전제조건

전제조건 역시 아주 쉽습니다.

  • node, git 설치
  • github에 사용자이름.github.io 리포지토리 생성
    다음으로…

Hexo 설치

$ npm install -g hexo-cli

블로그 생성 및 로컬 실행

$ hexo init blog
$ cd blog
$ npm install
$ npm install --save hexo-deployer-git

_config.yml 수정

내 사이트에 맞게 설정을 찾아 변경해줍니다.
자세한 사항은 공식 문서를 참조하세요.

로컬 실행

$ hexo server

위 명령어를 입력하면 http://localhost:4000/ 으로 접속이 가능합니다.

배포

$ hexo deploy --generate
INFO  Deploy done: git

이제 시작했으니 반은 했다는 기분으로 또 한동안 놀아야겠습니다. 헤헷!
