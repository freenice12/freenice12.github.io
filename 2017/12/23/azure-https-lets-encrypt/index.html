azure-https-lets-encrypt

2017-12-23
azure, letsencrypt

왜 잘 돌아가던 HTTPS 앱을 삭제하게 되었나.

이전에 개인 프로젝트를 하던 사이트는 IasS로(아마도?) 각종 환경을 CLI로 수정할 수 있었기 떄문에 큰 문제없이 시간만 투자(이게 문제지…)하면 그래도 원하는 것들이 적용되는 정도였습니다.

하지만 이번에 개인 프로젝트를 접고 WP 웹앱을 사용하기 위해 과감히 삭제 후 설치하게 되었습니다.

Azure - 웹앱

웹앱을 설치한 후 기본적인 설정을 마치고 사용자 도메인을 연결하였습니다.
여기까지 잘 작동함을 확인했지만, 기존에 Let`s encrypt를 사용해 항상 HTTPS로 접근했었기 떄문에 사용자 도메인으로 연결하게 되면 크롬 틴구가 불평을 했습니다.

위 메시지의 설명에는 해커가 나쁜짓 하는걸수도 있다~ 라는 문구가 있었지만, 그게 난데? 라는 자괴감이…

그래서 어쨋든 다시 Azure 웹엡에 렛츠인크립트를 설치해보기로 결정합니다.

삽을 들다.

구으글 신에게 여쭸습니다. “이거 웹엡에도 렛츠인크립트가 되나요?”
답이야 뻔하게 “당연하지 ~ 이런 문서들을 봐~” 하면서 많은 문서들이 나열 되었습니다.
대충 뭐가 필요한지는 알게 되었지만… 정작 제가 필요로 하는 순서인

  1. 웹앱 생성
  2. Azure Active Directory(AAD) 앱 생성
  3. 렛츠인크립트 추가

가 아니라
AAD 앱 생성 후 웹앱 연결인 반대의 순서로 설명이 되어 있었습니다(영어니까 잘 몰라요…).

뭐 암튼 필요하다는 목록에는

  • Tenant
  • SubscriptionId
  • ClientId
  • ClientSecret

등이 있었고, Azure도 모르고 용어도 모르는 저는 울 수 밖에 없었습니다. 몇몇 문서에서 설명해 주는대로 최대한 Azure portal에서 해결해보려고 버둥거렸습니다.
역시나 실패하고 우리의 Azure 한국 사용자 그룹에 문의를 날렸습니다.
이곳에서 엯촋님 유정협님이 답변을 달아주셨고, 설명은 매우 간단했습니다.

뭐 어쨋든 일단 필요한 값들을 얻는 방법을 한번 정리를 하자면…

  • Tenant: AAD > 속성 > 디렉토리 ID
  • SubscriptionId: 생성한 웹앱 > 개요 > 구독 ID
  • ClientId: AAD생성앱 >> 응용 프로그램 ID
  • ClientSecret: AAD생성앱 > 키 > 암호 생성 (기한 주고) > 등록하면 키 생김

이렇게 필수요소를 정리하고, 리소스 그룹 이름만 알면 모든 준비는 끝(일꺼라 생각했음…)입니다.

뭐여… 뭐가 자꼬 틀리데!

뭐 임마. 줘야하는 파라메터는 다 넣어줬잖아! 그만 불평하고 HTTPS연결이 되도록 일하라고!

아… 뭐 권한이 없다고 하는데 도리가 없었습니다. 권한을 줄 수밖에…
하지만 저는 무슨 권한이 필요한지 모르니까 무조건 다 줬습니다.
몇 일간 삽을들고 씨름하는 바람에 AAD에 만든 앱은 4개가 되었고 권한은 스크롤을 해야할 많큼 쌓여만 갔습니다(여윽시 멍튱이는 손발이…).
암튼 이 권한 저 권한 막 다 때려주고는 기도를 하면서
Azure Lets encrypt - Authentication Settings의 빈 칸을 마구 채워 넣습니다.
그러자 오류 메시지가 바뀌는 기적을 체험하였고, 그 메시지는 너 앱 설정 수정할껴? 라는 뉘앙스로 물어오는듯 하였습니다.
그러자 기적처럼 맨 아래에 Update Application Settings 체크박스가 눈에 띄였고, 체크해준 후 NEXT 버튼을 클릭하자… 팬티를 지를 수 밖에 없었습니다(소리 벗고).

힝~ 속았지~!?

다음 화면을 보고 매우 기뻣지만 긴장으 끈을 놓지 않았습니다.

이 페이지에는 뭐 글씨밖에 없고 사용자 입력을 받는 부분이 없어서 안심하고 NEXT를 눌렀습니다(오예 꿀~).

이 페이지에는 제가 사용자 도메인으로 등록해놓은 주소와 이메일을 입력하는 칸이 있었습니다.
(오예 내가 등록해 놓은 도메인이 보인다!!) 기뻐하며 이메일 주소를 입력하고 Request and Install certificate 버튼을 눌렀씁니다!
역시나 오류! ㅎㅎ 도메인이 보이는것에 안주하지 말고 살포시 클릭을 한번 해주라는 귀여운 경고를 맞이했습니다.
그래 까짓거 클릭해준다. 클릭 후 다시 Request and Install certificate 버튼을 눌렀씁니다!
와!! 크롬 탭에 골뱅이가 막 돌기 시작했습니다. 뭔가 될 듯한 이 긴장감!!!

그런데 갑자기 또 에러 페이지가????

403이면 권한인데… 또 Azure 한국 사용자 그룹에 유정협님께 도움!을 요청했습니다.
역시나 서비스 프린시팔(이름부터 참…) 권한이 아직 없다는 이야기를 들었습니다.

어?! 다 줬는데?!!?!?!?!!
다시 심기일전 한 후에 필요없는 것들(AAD 앱을 하나만 남김, 권한 줫던거 다 삭제)을 제거한 후에 심기일전 하며 클릭(이라고 쓰고 기도라고 읽는다)질을 해댔습니다.

그런데 삭제하던 도중 마지막 남은 AAD 앱에 기여자 권한만 빠져있는게 확인되었습니다(제길슨…). 낼름 기여자권한을 떠넘긴 후에 다시 Request and Install Certificate 페이지를 방문했습니다.

여정의 끝

이번에도 안되면 저녁에 정협님이 돌아와서 도와주시겟다고 했기때문에 모든걸 내려놓고 최대한 기도(라고 쓰고 클릭이라고.. 아니 실제 기도였잖아!?)를 했습니다.

Request and Install certificate 버튼을 눌렀(이라고 쓰고 기도라고 읽는다)씁니다!
또 설레이는 골뱅이가 돌아가고, 마침내 Success라는 단어와 마주하게 되었습니다.

이게 뭐라고 이렇게 절 힘들게 했는지… 서비스 프린시팔(이름부터 참…) 권한이 저를 한참 힘들게 했습니다.

다시 한번 최종적으로 각 키들을 어떻게 가져왔는지 어떤 기도를 했는지 정리한 후 마무리 하겠습니다.

마무으리

  • Tenant: AAD > 속성 > 디렉토리 ID
  • SubscriptionId: 생성한 웹앱 > 개요 > 구독 ID
  • ClientId: AAD생성앱 >> 응용 프로그램 ID
  • ClientSecret: AAD생성앱 > 키 > 암호 생성 (기한 주고) > 등록하면 키 생김
  • 리소스 어쩌고 하는것들은 리소스그룹 이름을 똑같이 넣어주세요!
  • 구독 > IAM > 독자 추가 > AAD 생성 앱 이름 검색 후 추가 (기여자로!)
    마지막 놈이 핵심입니다.
  • Update Application Settings 체크박스
  • 커스텀 도메인 클릭하기.

아래는 유정협님이 정리해주신 원문입니다(프린시팔 녀석만 잘 알았어도…욕이 아닙니다.ㅠㅠ).
페이스북 한국 Azure 사용자 그룹의 원문을 보실 수 있습니다.

이런 밥아자씨 같으신 분 ㅎㅎ
