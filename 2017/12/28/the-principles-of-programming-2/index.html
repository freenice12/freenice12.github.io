아키텍처 비기능 요구사항

  • 변경의 용의성
  • 상호 운용성
  • 효율성
  • 신뢰성
  • 테스트 용이성
  • 재사용성

비기능 요구사항에서는 보안관련 문제를 먼저 다룹니다. 배포 후 실제 프로그램이 작동할 때 3가지 측면에서 위험성을 이야기합니다.

  1. 권한이 없는 사용자의 접근 차단(기밀성)
  2. 권한이 없는 사용자의 데이터 변경 및 삭제 차단(무결성)
  3. 권한이 있는 사용자의 접근, 데이터 변경, 삭제 등 원하는 기능 제공(가용성)

위 3가지 측면을 검증할 때 외부 툴이나 팀에 요청을 해 공격적으로 위험을 탐지해야 한다고 말하고 있습니다.

변경의 용의성, 상호 운용성

변경의 용의성을 읽는동안 OCP생각이 계속났습니다. 간단해야 하고 응집(기능 등이)해 있어야 한다는 원리입니다. 상호 운용성은 이 후에 나올 UNIX의 철학과도 맞닿아 있는 개념입니다. 연계되는 소프트웨어를 대비하고 효율적으로 개발(바퀴의 재발명 NONO~!)하자는 취지입니다.

효율성, 신뢰성, 테스트 용이성, 재사용성

효율성은 시관(프로그램의 응답성)과 컴퓨팅 자원을 최대한 효율적으로 활용하자는 이야기를 하고 있습니다. 토론회의 주최자는 예전에 이런 느낌의 말을 한 적이 있습니다.

앞으로 컴퓨팅 자원이 좋아진다고 대충 개발한다는 생각은 너무 안일한 생각이다. IOT나 스마트폰(구형) 개발을 생각해 보라. 이는 더 안좋은 자원에서도 효율적으로 작동하는 프로그래밍이 여전히 유효하다는 이야기다.

그래서 결론은 컴퓨팅 자원이 좋다가도 좋지 아니하니까 정신 똑바로 차리고 개발하자는 이야기.

신뢰성에 대해서는 결함 허용력과 견고성을 이야기 하는데 이는 주로 Akka가 밀고있는 기조랑 겹치는 느낌을 받았습니다. Akka는 어떻게 실패를 허용하며 실패 시 어떤 동작을 할지. 이를 통해 어떻게 프로그램이 종료되지 않고도(아마도 스스로 복구하며) 계속 작동할지에 대한 이야기를 화두로 이야기 하고있습니다(당연히 예제 외에는 써본적이 없습니다… T_T). 혹은 넷플릭스의 Zuul(Hystrix - fallback, circuit 등)을 알아볼때 어떤식으로 장애를 다룰지, 장애시 어떤 결과를 보여줘 사용자에게 장애라는 사실을 알리지 않고도 유연하게 작동할지 등을 공부했던 내용이 막연히 떠올랐습니다(Akka와 마찬가지).

테스트에 대해서는 궁금한점이 많지만 당장 질문을 할 수 없기 떄문에 약간의 경험(없잖아?)을 바탕으로 내용을 풀어보겠습니다. 테스트야 말로 중복의 온상지이며 리팩토링을 경험할 최적의 분야가 아닌가 싶습니다. 저는 Puppeteer라는 노드 라이브러리르 이용해 테스트를 작성(시도)한 적이 있습니다. 로그인 부터 A 페이지의 기능 여러개 중 1개를 테스트하려고 했는데, 이 간단한 테스트를 위해서도 굉장히 많은 코드를 필요로 했습니다. 테스트 해야할 기능이 많아질 수록 코드가 더욱 많아질 것은 자명한 일이었습니다. 보통 프론트 개발자 분들은 어떤 식으로 테스트를 하는지 궁금했습니다. 물론 단위 테스트와 Puppeteer를 이용한 테스트는 느낌이 다르지만, 결국엔 둘 다 해야하는 테스트이기 떄문에 이 부분을 꼭 물어볼 생각입니다. 아무튼 테스트 하기 용이하지 않다!

재사용성도 마찬가지로 궁금한 것이 있는데 먼저 책이 이야기하고자 하는것은 먼저 만든걸 활용하자 플러그인 아키텍처!, 재사용 가능한 모듈을 만들고 사용하자 입니다. 지금 프로젝트에서는 ES6 미만의 문법을 사용하기 때문에 또 기술적 제약이 많이 있어서 import, export등을 사용할 수 없습니다. 동료(또 소환)는 모든 페이지에 모든 라이브러리를 소환해 사용하고 있는데 제가 생각할 때는 심각한 낭비라고 생각했습니다만, 별다른 대안이 떠오르지 않았습니다. 매 페이지마다 스크립트 태그로 불러오는게 좋겠지만 이걸 싫어하는 동료를 설득할 자신이 없었습니다(이것도 물어봐야겠다). 그리고 이어지는 내용은 모듈화 할때 일반화를 거치게 되는데 어디서도 통용되는 일반화를 할때는 구현 난이도가 3배(책은 3배라지만 매우) 어려워 진다고 합니다. 아무튼 SI(싫어!)업계에서 말하는 공통 모듈도 여기에 한 발 걸쳐있다고 생각하는데, 역시나 기저에 돌아가는 코드가 옳은 코드다.를 깔고 있는 분들이 많아 신잎인 저는 잎파리만 닦고 앉아있습니다.

물론 위 내용은 비기능 요구사항이기 때문에 기능적 요구사항과 마찬가지로 매우 중요하지만 놓치기 쉬운 부분들이라 더 신경쓰지 않으면 곤란한 부분입니다.

7가지 설계 원리

이어지는 내용은 7가시 설계 원리에 대한 내용입니다. 여기서도 이 책의 오타라고 생각된 경험칙이란 단어가 등장하는데(찾아보니 관찰과 측정을 통해 얻은 법칙 이란 뜻) 무지의 얼굴 붉힘 ㅠㅠ, 암튼 제가 제일 받고싶은 코드 리뷰의 내용이 나옵니다.

제가 코드 리뷰에서 가장 중요하다고 생각하는게 컨벤션 같은 친구들은 최대한 기계가 맞춰(지적?)야한다!입니다. 이 부분도 사실 팀에서 정했으면 따르는게 맞는데 무소의 뿔같은 분들은 이야기하면 굉장히 기분 나빠하는 경험(지켜봄.. 말할 짬은 안되니까)을 했습니다. 암튼 내용은 아래와 같습니다.

7가지 원리:

  • 단순 원리
  • 동형 원리
  • 대칭 원리
  • 계층 원리
  • 선형 원리
  • 명증 원리
  • 안전 원리

단순 원리

IT 첫 회사에서 많이 들었던 말이 “메소드는 한 가지 일을 해야 하며 10줄 정도로 작성하는것이 좋다(보통은).”는 말이었습니다. 물론 모든 메소드를 10줄이내로 작성하는건 사치이면서 복잡성이 증가되는 느낌이고 저도 최대한 이 말을 따르고 있지만, 정작 제대로 지키면서 코드를 작성하기는 힘들다는 생각을 할 때가 많습니다. 이런 단순하고 적은량의 코드로도 잘 작동하고 자연스러움이 묻어난다면 높은 수준의 개발자가 아닐까 생각합니다.

동형 원리, 대칭 원리

동형 원리는 최근에 복붙(?) 코드가 많아지면서 항상 틀을 정하고 코드를 작성 및 수정하는데, 흐름이 있습니다. 프레임워크를 사용한다면 제공되는 라이프 사이클을 이용할 수 있으니 한결 수월하겠지만… 현재는 최초 파일 호출시 진입점으로 동적 테이블 작성 후 후처리로 모든 로직을 모으고 있습니다. 업무 특성상 최초 동적 테이블 생성 후 후 처리에서 작업을 해야 하기 떄문에 이런 틀을 가지고 작업합니다. 이 틀에서 벗어난다면 복잡한 코드일테고 더 집중해서 봐야한다는 신호일 수 있습니다. 그리고 버그의 온상이 되겠죠.

대칭 원리는 get이 있으면 set이 있다거나 클릭 이벤트로 hidden을 줬다면 반드시 show를 구현한다거나 하는 원리를 말하고 있습니다. 예측이 가능해서 쉽게 버그나 결함을 찾을 수 있다는 장점이 있습니다.

계층 원리, 선형 원리, 명증 원리, 안전 원리

계층 원리는 이해하기 힘든 원리인데 계층이 있으면 읽기 쉽다, 상위에서 바라보듯 계층 구조로 작성하자는 이야기를 하고 있는데 토론회에서 자세히 들어봐야할것 같습니다.

선형 원리는 흐름대로 읽히는 코드에 관한 내용인데, 최근에는 Async라던지 흐름의 분기(if 혹은 switch)가 있으면 코드를 읽기 힘들고 단순하게 작성하기가 힘들다(읽기도 힘들고)는 내용입니다. 이전 세미나였던 코드 스피츠 세미나에는 if 등의 흐름을 분기시키는 문구로 제어할 때 반드시 의도를 나타낼 수 있는 코드를 작성하는게 좋다는 취지의 이야기를 들었기때문에, 의도가 있다면 필수라면 꼭 넣어주되 필요없는 내용인지 확인 후 최대한 깔끔(삭제)하게 유지하는 융통성이 필요한 것 같습니다.

명증 원리는 로직의 명확함을 중시하라는 내용입니다. 최근 저도 몇 번의 코드 수정끝에 논리적으로 놓친 코드에서 버그를 유발했던 기억이 있습니다.

위 코드와 비슷한 부분이었는데 논리적으로 최초 someMethod를 호출하는 부분에 도달했다면 master도 아닐 뿐더러 isMaster라는 파라메터 또한 불필요한 변수가 됩니다. 하지만 이를 발견한건 버그 수정 요청이 들어온 후였고, 매우 부끄러웠던 기억이 있습니다(정신 차리자?! 틴구야?!).

마지막으로 안전 원리 역시 이전 코드 스피츠 강의에서 들은대로 실천하고 있습니다. 코드는 반드시 의도를 표현해야 한다는 명제 등입니다.

등의 내용입니다.

UNIX 사상

UNIX는 오래 전부터 살아 남은 멋진 작품이고, 이는 많은 프로그래머들에게 지침과도 같은 사상과 철학을 이야기 해주고있습니다.

모듈화의 원칙, 명확성의 원칙, 구성의 원칙

모듈화나 명확성은 비슷한 이야기 같습니다. 예를들어 함수형 프로그래밍에서 말하는 부수효과가 없는 함수들을 떠올려본다면, 비슷한 느낌일 것입니다. 굉장히 세부적이고 함수는 특정 행동만 한다고 하면 명확한 모듈이 됩니다. 이를 구성하는것이 핵심인데 쉘 스크립트에서 필터(파이프라이닝)를 사용하는 문법을 떠올리면 되겠습니다. 프로그램의 아주 근본이 되는 원칙들입니다.

분리의 원칙

정책을 반영하는 것은 프로그래머보다는 운영자(사용자)일 확률이 높습니다. 프로그래머가 아니라면 일반적인 문서에 규칙을 제공해주고 이를 수정하게 하는편이 프로그램 구동 문제에도 더욱 좋은 방향일 것입니다. 즉 메커니즘(코드)과 정책(관련 변수 등)은 분리되는 것이 모두에게 좋다고 말하고있습니다.

단순성의 원칙, 절약의 원칙

결코 코드로 자신을 내보이려 하지 말고, 잘못된 자부심을 가진다면 코드(프로그램)는 필연적으로 복잡해진다는 이야기입니다. 항상 명심해야 할 주제입니다. 이와 비슷하게 절약이라고 하는것은 불필요하게 코드를 작성하지 말고 종이를 아까워하듯 코드도 아까워하며 섬세한 논리(생각)를 표현하라는 의미입니다. 코드가 커(복잡)지는 것을 막고, 애매한 방식으로 코드를 키우(덧 붙이는)는 것을 멀리하라는 이야기도 하고 있습니다.

투명성의 원칙, 안정성의 원칙

  • 투명성: 프로그램의 동작이 쉽게 파악될 수 있을 것
  • 개시성: 프로그램의 내부 상태를 볼(혹은 감시) 수 있을 것

위 두가지를 의식한다면 디버깅 관점에서 좀 더 빠른 문제 해결에 도달할 수 있을것입니다. 우리가 로깅을 하는 이유는 개시성을 의식해서이고 좀 더 직관적인 UI/UX를 하려는 이유는 투명성을 위해서입니다. 이런 구분법이 있는지 모르고 있지만 역시나 대부분의 프로그래머들이 실천하고 있는 원칙이라고 생각하고 있습니다.

위에서 말한 모듈, 명확, 단순, 투명 등의 원칙을 잘 지킨다면 프로그램은 견고할 수밖에 없습니다. 혹은 결함이 있다 하더라도 이를 해결하는 속도는 그렇지 않은 프로그램의 문제를 해결하는 시간 이하 일것입니다.

표현성의 원칙

이해가 잘 가지 않는 부분인데, 복잡함은 데이터에 모으고 자료구조를 복잡하게 할지 로직을 복잡하게 할지 선택해야 한다면 전자를 선택하라는 내용입니다.

충격 최소의 원칙

일반적인 흐름이나 예상에 따라야 한다는 내용입니다. 기존의 인터페이스가 작동하는데로 미묘한 차이도 없이 예상대로 작동해야 하며 기존에 존재하는 부분을 차용했다면 똑같이 동작해야 함을 의미합니다.

침묵의 원칙

불필요한 정보를 노출시킬 필요는 없고, 이런 노출에 의해 오히려 중요한 정보가 노출될 수 있는 위험성을 제거하자는 원칙입니다.

복구의 원칙

오류가 발생했는데 오류를 잘못 파악해 계속 실행되게 한다면 피해가 확대될 것입니다. 또한 오류가 발생되었을 때 가장 빨리 알 수 있도록 요란한 통지를 유도합니다. 입, 출력에 관해서도 받는건 자유로, 내보낼 때는 보수적으로라는 사고방식을 가지라고 합니다.

경제성의 원칙, 생성의 원칙

프로그래머의 시간은 소중하니까 낭비하지 말고, 될 수 있다면 자동적으로 처리할 수 있는 부분은 자동으로 처리할 수 있도록 코드를 작성하고 더 반복적이라면 코드 생성기를 통해 코드를 생성하라고 합니다.

최적화의 원칙

바르게 만들고 빠르게 만든다.라고 이야기 하고있지만, 가장 중요한건 아래 두 가지 원칙이 아닐까 합니다.

  • 아직 아니다
  • 그래도 아직 아니다!

어설픈 최적화는 복잡한 코드를 생성하고 결국 원하는 결과도 얻기 힘드니 역시 신중하고 꼼꼼하게 진행해야 할것 같습니다.

다양성의 원칙, 확장성의 원칙

더 좋은 방법을 계속 생각하고, 확장할 수 있도록 설계하라는 내용이 핵심입니다(자기 설명적인 데이터 형식?).

UNIX 철학

이 책의 내용과 마찬가지로 굉장히 보편적이지만 여지껏 살아남은 철학들입니다.

  • 작은것이 아름답다
  • 한 번에 하나의 작업
  • 즉시 프로토타입 진행
  • 효율보다 이식성
  • 데이터는 텍스트로
  • 레버리지 소프트웨어
  • 셸 스크립트 활용
  • 대화형 인터페이스 회피
  • 필터화

작은것이 아름답다, 한 번에 하나의 작업, 즉시 프로토타입 진행

UNIX의 사상에서 이미 확인했듯, 아주 작은 부품들을 만들어 가는 그리고 그것들은 온전히 하나의 작업에 집중하도록 만드는 철학을 가지고 있습니다. 이 철학을 바탕으로 부품들의 조합을 통해 소위 말하는 MVP(Minimum Viable Product)를 만들 수 있습니다.

모임의 주최자가 한 이야긴지는 잘 기억이(ㅠㅠ)나지 않지만, 우리가 알고있는 MVP이미지(퀵보드 > 스쿠터 > 자동차)는 잘못된 것이라고 볼 수 있다는 이야기를 들었습니다. 퀵보드가 스쿠터가 되거나 스쿠터가 자동차가 되는것은 이전것을 버리고 새로 만드는 개념이라 어울리지 않는다고 이야기합니다. 아무튼 의미는 전달된 것 같으니 잘못된 구체적 예를보고 오류를 범하지 않아야 합니다.

효율보다 이식성

이는 그동안 프로그램에 국한해 생각했던 UNIX 사상과도 조금은 다른 이야기를 하고 있습니다. 하지만 이 장은 UNIX의 철학을 이야기 하는 장이기 떄문에 당연히 이식성이 우선 고려되어야 한다고 생각했습니다. 물론 대부분의 프로그래머(저 같은 웹 쫄병)는 해당이 되지 않는 이야기 같습니다. 우리는 효율입니다! 효율!

데이터는 텍스트로

텍스트는 사람이 읽기 쉽기 때문에 (읽기)쉬운 코드를 작성하라는 말과도 일맥 상통하는 이야기입니다. 게다가 UNIX(류의 운영체제)에서는 필터(파이프라인)를 통해서 텍스트 데이터들이 입력과 출력으로 이동하고 결과역시 콘솔에 텍스트로 출력됩니다. 굉장히 인간 친화적입니다. 물론 때에따라 적용할 수 없겠지만, JSON등의 텍스트 형식이나 기본 데이터 객체의 toString 메소드를 반드시 구현해서 확인하는 습관(저는 있다고 생각합니다)이 있기를 바라는 주제입니다.

레버리지 소프트웨어, 셸 스크립트 활용

이 주제 역시 효율 극대화를 이야기 하고있습니다. 가장 마지막에 수작업을 자동화 하라는 이야기를 하고 있는데, 이 주제에 가장 부합하는 내용인것 같습니다. 바로 셸 스크립트를 활용하라고 하는데 셸 스크립트가 조금 어렵(쓰기는 쉬워도 읽기는 어렵)다는 의견이 많아 최근에는 파이선이나 스칼라 등을 이용해 스크립팅하는 분들을 많이 목격했습니다. 조금 더 수월할것 같지만, 아직 뭐가 필요한지 모르기 때문에 시도해보지는 못했습니다.

대화형 인터페이스 회피, 필터화

UNIX의 많은(아마도 전부?) 명령들이 입, 출력을 가지는 일종의 함수 개념인 것으로 알고있습니다(물론 아니겠죠? TT). 이는 철학에 근거해 대화형 인터페이스(아마도 말이 길어지면 논지가 흐려지니까…)보다는 단답(혹은 명령)형식의 인터페이스를 기반으로 하고있는것 같습니다. 여러번 등장한 필터화에 유리하고 병렬처리나 부수효과가 모아진다는 장점등 제가 설명할 수 없는 범위까지 장점들이 존재하는것 같습니다.

UNIX 사상, 철학

이로써 3장이 끝났습니다. UNIX의 사상과 철학에는 배울점이 많지만 굉장히 추상적(일반적)이라 구체화해서 이해하기는 조금 어려운 부분도 존재합니다. 이를 체화한 동료를 만난다면 당연히 자연스럽게(너무 자연스러워 모를수도 ㅠㅠ) 접할 수 있는 내용이겠지만, 정신차리지 않고 시간을 흘려보내면 결국 남는것은 2018년 회고일뿐입니다. 라고 생각합니다.
