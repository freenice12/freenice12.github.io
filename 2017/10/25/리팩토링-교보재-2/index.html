리팩토링-교보재-2

오늘은 리팩토링 교보재 라는 제목으로 글을 쓰는게 정확한가에 대한 논의를 했습니다. 결론은 게시하는 내용은 리팩토링이라기엔 아리송한 부분이 많다.였고 그럼에도 불구하고 그냥 이어 나가기로 했습니다.

정답을 게시하는 것은 아니고 주관적인 생각을 쓴것이고, 많은 부분 틀릴 수 있습니다. 얼마든지 피드백을 주세요(달라고만 했음…).

Spring Framework - 필드 뭣이 중헌디?!

오늘의 주제는 Spring Framework - 필드 뭣이 중헌디?! 입니다.

스프링의 기본 스코프는 싱글톤이고, 필드는 스레드 세이프 하지 않습니다. 이 내용을 골자로 하는 실무에서 잘못 적용된 사례를 수정해보도록 하겠습니다. 사실 더 좋은 방법을 알고싶습니다.

아래는 컨트롤러 메소드를 구현했습니다.
3초 짜리 로직을 담은 메소드와 1초짜리 로직을 담은 메소드입니다.

정상 작동하는 시나리오 - 특정 상황

서로 간섭없이 각각의 api를 호출합니다.

비정상 시나리오 - 일반 상황

보통의 상황에서 스프링을 사용하고 웹 어플리케이션을 작성했다면 API 호출이 뒤섞이고 난무할 가능성이 높습니다(개발자가 로컬로 혼자 테스트 하는 환경보다는).
시나리오를 가정하고 3초짜리 호출이 요청 된 후에 끝나기 전에 1초짜리 호출이 요청되는 경우를 테스트 해보겠습니다.

이유는 인터셉터 구현부에서 필드를 사용한 startTime 변수가 스레드 세이프하지 않아서 입니다. 레퍼런스 문서에도 인터셉터는 스레드 세이프하지 않다고 주의하라는 내용이 나와있습니다.
그럼 어떻게 해결해야 할까요?
저는 2가지 방법을 생각했습니다(하나는 또 틀렸…TT).

  • ThreadLocal을 사용하자!
    • 가장 처음 생각했고
    • 자바 1.2부터 존재했으며
    • 스프링 시큐리티에서도 사용하니까?!
    • 더 많은 이유가 있겠지만 제가 몰라요…
  • 세션을 이용하자?!
    • 응 아니야~
    • 틀려!

ThreadLocal로 해결!

먼저 제가 아는 최선의 방법으로 해결했습니다.
딱 메소드 시간을 측정한다는 목적에만 집중했습니다.
다른 문제나 더 좋은 방법을 알려주세요!

단순히 무쓸모한 필드를 없애고 ThreadLocal을 사용했습니다.

결과는 당연히 정상입니다.

session으로 해결

각 api마다 시간을 저장해서 종료 후 호출하는 방식을 사용했습니다. 결과도 물론 이상없이 출력됩니다.

하지만 같은 API호출을 반복(키 관리 문제)하는 등의 문제로 기존의 문제점을 해결했다기 보다는 그냥 똑같이 더러운 코드를 생성한 느낌입니다. 위에서 가정한 시나리오 상에는 정상작동 합니다(틀린거라고). 이는 소위 제 컴퓨터에선 되는데요?, 로컬에선 됩니다. 등의 문제를 야기하는 원인이 됩니다.나빠

그래서 결론은?

다시 이야기 하자면

  • 스프링은 싱글톤이다.
  • 스레드 세이프인지 신경쓰는 후로그래머가 되자.
  • 기술은 상황에 맞게 쓰자.
  • 문제 해결 능력(삽니다)을 기르자.
  • 변수를 막 쓰지말자.

감사합니다!

끗!
